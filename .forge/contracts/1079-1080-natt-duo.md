# Spec: Natt-duoen — Nattkjøreren (#1079) + Morgenbriefen (#1080)

**Issues:** #1079 + #1080 (epic #1073, nr. 6 og 7 — bygges sammen)
**Branch:** claude/1079-1080-natt-duo

## Problem

Backloggen beveger seg bare når et menneske starter en økt, og loopene som nå finnes har ingen felles kvittering eieren kan stole på. Nattkjøreren gjør eier-merkede issues til ferdig-verifiserte draft-PR-er mens eieren sover; Morgenbriefen er den daglige, verifiserte handlingslista som gjør at ingen loop kan feile stille.

## Research Findings / rammebetingelser

- **Cloud-modellen forenkler originaldesignet:** hver routine-kjøring får fersk klone i isolert VM → hele det lokale preflight-/rogue-detektor-/låsefil-maskineriet fra issue-teksten bortfaller. Gjenstår som reelle vern: kø-disiplin, konvergensregler (#1077), aldri-merge, fail-closed-utfall.
- Routines kan pushe `claude/`-brancher og åpne PR-er under eiers identitet (verifisert i #1073-kommentar); kontrakter ligger som issue-kommentarer (forge-konvensjonen) og er dermed synlige i fersk klone.
- **Staging-e2e fra skyen:** krever staging-secrets i routine-miljøet + nettverkstilgang til staging-hosten. v1 er dette VALGFRITT: med secrets kjøres `npm run e2e:gate`; uten → PR-en får `needs-manual-qa`-label (eksplisitt fail-closed-utfall, aldri stille grønt). Full #1076-portkjøring (preview-driving) skjer i interaktive økter.
- Loop-drift-issuet #1110 (pinned) er opprettet som felles heartbeat-/brief-tavle. CI-vakta poster IKKE heartbeat i v1 (24/døgn = støy; liveness sees på routines-siden) — dokumentert avvik fra opprinnelig #1080-design.

## Design

### docs/loops/nattkjoreren.md (ny)

1. **Kø:** `autonomy:ready`-merkede åpne issues; plukk ÉN (eldste). Tom kø → heartbeat «ingen kø» på #1110, ferdig (suksess).
2. **Per-issue preflight (fail-closed):** issuet har forge-kontrakt som issue-kommentar (── ellers `autonomy:blocked` + norsk kommentar, ta neste — maks ett forsøk til, så avslutt natten); `npm ci` + gates grønne på fersk main FØR bygging (rød main → stopp hele natten + sjekk at CI-vakt-varselissue finnes).
3. **Bygg:** forge:auto-disiplin mot kontrakten — bygge→evaluere→fikse til ACCEPT, med #1077-konvergensreglene (runde-historikk committes, maks 5 runder, strategibytte, eskaleringsmal ved ikke-konvergens).
4. **Verifisering:** alle gates + `npm run e2e:gate` hvis staging-env finnes; ellers `needs-manual-qa`-label.
5. **Lever:** DRAFT-PR med `Closes #N`, `autonomy:review`-label, norsk PR-kommentar (hva, hvorfor, bevis). ALDRI merge. Ikke-konvergens → draft-PR + eskaleringsmal + `autonomy:blocked`.
6. **Heartbeat:** ALLTID én avsluttende kommentar på #1110 (utfall: bygd/blokkert/tom kø/avbrutt), uansett hva.
7. **Budsjett v1:** maks 1 issue per natt til eieren eksplisitt øker.

### docs/loops/morgenbriefen.md (ny)

1. **Delta:** finn forrige brief-kommentar på #1110; rapporter kun det som er nytt siden.
2. **Verifiser FØR inkludering:** hver påstand sjekkes med gh (PR-checks faktisk grønne, issue faktisk åpent/lukket, label faktisk satt). En påstand som ikke verifiserer → rapporteres som loop-feil i stedet.
3. **Format:** én handling per linje med lenke («Godkjenn PR #X — evaluate ACCEPT + gates grønne», «Svar A/B på #Y»), gruppert: Trenger deg nå / Skjedde i natt / Loop-helse.
4. **Heartbeat-vakt:** Nattkjøreren og Dok-avstemmeren (når due) skal ha heartbeat siden forrige brief — mangler den to kjøringer på rad → infra-issue + varsellinje øverst.
5. **Tom natt:** én linje («ingen aktivitet — loopene har heartbeat»). Stillhet fra briefen selv = feil; eieren sjekker routines-siden.

### Små oppdateringer

- `docs/loops/ci-vakta.md`: heartbeat-avsnittet konkretiseres (peker på #1110, presiserer at CI-vakta selv IKKE poster heartbeat i v1).
- `docs/loops/dok-avstemmeren.md`: steg 4-heartbeat peker på #1110.

## Key Decisions

- Lokal preflight/rogue-detektor/låsefil DROPPET — cloud-klonen gjør dem overflødige; nevnes som avvik i closing-kommentarene.
- CI-vakta uten heartbeat i v1 (støy); liveness via routines-siden. Revurderes hvis en stille CI-vakt-død faktisk inntreffer.
- Én natt-issue av gangen til eieren øker budsjettet — tillits-opptrapping per epic-planen.
- Push-varsel til telefon utsatt (GitHub-notifikasjoner dekker v1; PWA-push er app-infra, eget issue hvis ønsket).
