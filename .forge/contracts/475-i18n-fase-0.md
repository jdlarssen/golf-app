## Locked architecture (from epic contract + issue #475 — do not re-litigate)

- `next-intl` (v4.13.x), App Router, `localePrefix: 'as-needed'`.
- `routing.ts` lists ONLY `['no', 'en']` in Phase 0 (gd/ga later — listing them
  now breaks message-loading on missing catalogs).
- Negotiation precedence: `users.locale` → `NEXT_LOCALE` cookie →
  `Accept-Language` (no/en only) → default `no`.
- Norwegian URLs stay exactly as today (unprefixed). `/en/<path>` resolves.
- Slug segments stay Norwegian. No route renaming.

## New constraint discovered 2026-06-10 (post-dates the epic contract)

The app shipped `cacheComponents: true` in #538 (v1.108.6, 2026-06-10) — three
days AFTER the epic contract was written. next-intl's default request-config
reads a middleware header at render time, which is request-time data and
breaks PPR static shells under cacheComponents (upstream: amannn/next-intl#1493,
official support pending).

**Resolution (locked for this build):** use the documented Next 16.2+ pattern
(Aurora Scharff, updated 2026-03-24; our Next is 16.2.6):

1. Enable `experimental: { rootParams: true }` in `next.config.ts`.
2. `i18n/request.ts` resolves locale via `await locale()` from
   `next/root-params` — a route param, not request-time data, so it becomes
   part of the cache key and PPR shells prerender per locale.
3. `generateStaticParams()` on `app/[locale]/layout.tsx` returns both locales.
4. NO `setRequestLocale()` calls needed anywhere (the root-params read replaces
   the header read entirely).

**Hard gate:** PPR shell coverage must not collapse. Baseline build output is
captured pre-change (`/tmp/baseline-build.txt`); post-change build must show
the same set of routes keeping partial-prerender (◐) status, modulo the
expected `/[locale]/` path prefix. If after honest attempts (max 3 build-fix
rounds on the same failure) the pattern can't hold PPR, STOP and flag to the
user — do not silently ship a fully-dynamic app and do not remove
`cacheComponents: true`.

## Design decisions (Claude's discretion, recorded)

- **`users.locale` read cost:** proxy reads `users.locale` from DB ONLY when
  the request has no `NEXT_LOCALE` cookie (≈ first visit per device), then
  persists the resolved locale to the cookie so subsequent requests skip the
  DB read. Honors the precedence chain for fresh browsers (the acceptance
  test) at ~zero steady-state cost. Known limitation, documented in code: a
  device with a stale cookie won't see a cross-device `users.locale` change
  until the cookie is refreshed — the Phase 1 Profil toggle updates both DB
  and cookie, which keeps same-device behavior exact.
- **Proxy composition order** (follows next-intl's official Supabase example):
  (1) classify path public/protected in code (matcher now matches ALL pages,
  including login/legal/signup — required because `as-needed` rewrites
  unprefixed URLs via middleware; the old matcher exclusions would 404 every
  public page); (2) run `createMiddleware(routing)` → intl response (rewrite/
  redirect + locale cookie); (3) if protected: Supabase session refresh bound
  to the SAME response object (cookies merge), auth-gate redirect to
  locale-aware `/login` + `?next=` on no-user, `x-torny-user-id` header,
  `last_seen_at` fire-and-forget — all preserved.
- **Pure negotiation logic** extracted to `lib/i18n/resolveLocale.ts`
  (userLocale, cookieLocale, acceptLanguageHeader) → locale, so precedence is
  Type A-testable without the proxy.
- **`lib/i18n/format.ts`:** locale-aware `formatDate`/`formatTime`/
  `formatNumber`/`formatRelativeTime(value, opts, locale)`. For `'no'` it
  delegates to the existing hand-rolled `lib/format/{date,teeOff,relativeTimeNb}`
  helpers (their output is deliberately NOT raw Intl — trailing-dot fixes etc.;
  preserving exact output = no user-visible change). For other locales it uses
  `Intl.*` with the target locale. The `nb-NO` literals inside `lib/format/*`
  are the no-branch implementation, not "hardcoded call-sites"; the ~11 inline
  `'nb-NO'` call-sites in `app/**`/`components/**` migrate to the new util
  (server components get locale via `getLocale()`, client via `useLocale()`).
- **`app/manifest.ts` `lang`:** stays `nb-NO` (static file, one manifest for
  the PWA shell; revisit in a later phase). Noted per issue task 8.
- **Smoke-key proof:** `messages/{no,en}.json` get `common.appName: "Tørny"`;
  `app/[locale]/layout.tsx` `generateMetadata` reads it via
  `getTranslations` — catalog loading is proven at build time with byte-identical
  user-visible output.
- **Lint guard mechanism:** extend `.githooks/pre-commit` (same warn-only,
  added-lines-only approach as the existing AI-tells guard) to flag new
  hardcoded user-facing string literals (JSX text / Norwegian-letter strings)
  in `app/**` + `components/**`. Warn-only — the rollout is long and the
  existing 366 files would make a blocking rule unusable.
- **Files that stay OUTSIDE `app/[locale]/`:** `app/api/**`, `app/manifest.ts`,
  `app/icon.tsx`, `app/icon0.tsx`, `app/apple-icon.tsx`, `app/favicon.ico`,
  `app/globals.css`. Everything else moves under `app/[locale]/` via `git mv`.
  No root `app/layout.tsx` remains (next-intl official structure —
  `app/[locale]/layout.tsx` IS the root layout); `@/app/...` imports get the
  `[locale]` segment inserted mechanically.
- **Migration:** `supabase/migrations/0093_users_locale.sql` — `alter table
  public.users add column locale text;` (nullable, no CHECK; app-level
  validation). Applied via Supabase MCP during build (additive + inert until
  code reads it).

## Acceptance criteria (from issue #475)

- [ ] `npm run build` green under `app/[locale]/`; PPR shell coverage not
      collapsed vs baseline (hard gate above).
- [ ] All existing Norwegian URLs resolve unprefixed; `/en/<path>` resolves
      (empty EN catalog → Norwegian fallback OK).
- [ ] Negotiation: fresh browser `Accept-Language: en` → English routing;
      absent/`no` → Norwegian; `users.locale` overrides both (fresh-browser
      case; staleness limitation documented).
- [ ] Auth-gate redirect preserves locale + `?next=`; `x-torny-user-id` +
      `last_seen_at` still work.
- [ ] No remaining hardcoded `nb-NO` in migrated call-sites; `lib/i18n/format.ts`
      Type A tests pass.
- [ ] N-locale proof: stub 3rd locale = `routing.ts` array edit + catalog file
      only (verified, then reverted).
- [ ] `npx tsc --noEmit` green.
