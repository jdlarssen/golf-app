## Correctness verification (the skeptical checks)

- **Next.js 16 prop name `unstable_retry` — CORRECT.** Verified against
  `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/error.md`. The doc's
  canonical example (L25–31, L117–139) uses `unstable_retry: () => void` for `error.tsx` and
  `global-error.tsx`. Version History (L329): "`v16.2.0` — `unstable_retry` prop added." Installed
  Next is **16.2.6**, so the prop exists. The doc explicitly recommends `unstable_retry` over `reset`
  (L121: "re-fetch and re-render the error boundary's children"), which matches the contract's
  reasoning for a transient Supabase hiccup. Default-export signatures `{ error, unstable_retry }`
  are correct in all three files.

- **Boundary coverage — CORRECT and complete.** `app/[locale]/games/[id]/error.tsx` wraps the
  segment's nested children. The four scoring routes are all SERVER components doing Supabase reads
  (verified `(home)/page.tsx`, `leaderboard/page.tsx`, `submit/page.tsx`,
  `holes/[holeNumber]/page.tsx` — none are `'use client'`), so their throws are caught. The
  `holes/` route is actually `holes/[holeNumber]/page.tsx` (nested child) — covered. There is a
  `games/[id]/layout.tsx` and it DOES throwable work (`await getGameWithPlayers(id)` L24). Per the
  Next doc (error.md L96: error.js "does **not** wrap the `layout.js` … above it in the same
  segment"), that layout throw is NOT caught by the game boundary — it bubbles to
  `app/[locale]/error.tsx`, exactly as the contract's table (L20–21) and the file's JSDoc claim.
  `app/global-error.tsx` is at the app root (correct, since root layout = `app/[locale]/layout.tsx`)
  and catches root-layout throws. There are exactly 3 error files, no shadowing nested boundary.

- **i18n provider — CORRECT.** `NextIntlClientProvider` is present in `app/[locale]/layout.tsx`
  (L97), wrapping `{children}`. Both client route boundaries render under `[locale]`, so
  `useTranslations('error')` resolves at runtime. (In unit tests, next-intl is globally mocked
  against the real no.json, so no provider is needed there — documented in the test L5–6.)

- **global-error robustness — CORRECT.** Own `<html lang="no"><body>`, hardcoded Norwegian, inline
  styles (no dependence on globals.css, fonts, or the provider — appropriate since it replaces the
  root layout), 44px tap targets, a retry button (`unstable_retry()`) AND a home link (`<a href="/">`,
  a full reload, which is the robust choice when the router may be unmounted).

## Findings

- **F1 (LOW, non-blocking) — `app/global-error.tsx:73` adds ESLint errors.** `npx eslint
  app/global-error.tsx` exits 1: `@next/next/no-html-link-for-pages` on the `<a href="/">`
  (reported 20× by the rule). NOT a blocker because: (a) Next 16 `next build` no longer runs ESLint
  (`node_modules/next/dist/docs/.../06-cli/next.md` L103: "linting will be removed from `next build`
  in Next 16"), so the proven BUILD_EXIT=0 stands; (b) the repo's CI (`.github/workflows/ci.yml`
  L5–7) runs lint with `continue-on-error` precisely because of the same deliberate `<a>` pattern in
  `/legal/privacy`. The contract (K3) explicitly *specifies* `<a href="/">` here, and it is the
  correct robust choice for global-error (full reload when the router shell is gone), mirroring the
  existing deliberate exception. Optional polish: add an
  `// eslint-disable-next-line @next/next/no-html-link-for-pages` above L73 to keep the lint signal
  clean (matching how `/legal/privacy` is handled, if it disables there). Does not change verdict.

- **F2 (INFO, pre-existing convention, not a regression) — bare-`/` hrefs under non-default locale.**
  `ErrorScreen`'s `LinkButton` uses `SmartLink` (= plain `next/link`), not next-intl's locale-aware
  `Link`. For an English user on `/en/...`, `href="/games/${id}"` / `href="/"` are not locale-prefixed.
  This is the *same* pattern as the shipped, working `not-found.tsx` (`LinkButton href="/"`), and with
  `localePrefix: 'as-needed'` + proxy rewrite the bare path resolves to default locale `no`. Norwegian
  (default, unprefixed) users get exactly the right target. This is an established app-wide convention,
  not introduced by #680, so it is not a blocker for this contract. If locale-correct back-links for
  English users are ever wanted, that is a separate cross-cutting change (would touch not-found too).
